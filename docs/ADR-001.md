# ADR-001: Transactional Outbox Pattern for Kafka Event Publishing

**Date:** 2024-04-23  
**Status:** Accepted  
**Author:** Padmini Ishwarya Koduri

---

## Context

The Order Service is the core service of a food delivery platform. It handles order placement, status transitions, and coordinates with Payment Service and Notification Service via Kafka events.

The assignment requires publishing a Kafka event on every successful order status change. These events are critical — Payment Service uses them to initiate payment, Notification Service uses them to update the customer.

The core problem is that PostgreSQL and Kafka are two separate systems. There is no single transaction that spans both. A simple implementation saves the order to the database and then publishes to Kafka. This has a critical failure gap — if the service crashes after the database write but before the Kafka publish, the order status is updated but the event is never sent. Payment is never initiated. The customer's order is stuck. Downstream services have no idea the status changed.

The assignment also states that the status update endpoint will be called by an internal service with automatic retries — the same transition may arrive up to 3 times. The solution must handle duplicate events safely on both the producer and consumer side.

---

## Options Considered

### Option 1 — Direct dual-write

Save to DB then publish to Kafka in the same method.

```java
orderRepository.save(order);
kafkaTemplate.send(topic, payload); // crash here = lost event
```

**Problems:**
- No atomicity between DB write and Kafka publish
- Service crash between the two leaves inconsistent state
- Kafka being down causes order placement to fail entirely
- Lost events have no recovery mechanism — silent failure

**Verdict:** Rejected. Data loss is possible with no recovery path.

---

### Option 2 — Synchronous Kafka publish inside the DB transaction

Open a DB transaction, publish to Kafka inside it, then commit.

```java
@Transactional
public void updateStatus(...) {
    orderRepository.save(order);
    kafkaTemplate.send(topic, payload); // inside transaction
}
```

**Problems:**
- Kafka publish inside a DB transaction holds the DB connection open for the entire Kafka send duration
- If Kafka is slow or unavailable, the connection pool exhausts under load
- Still no atomicity guarantee — Kafka send can succeed but DB transaction can roll back

**Verdict:** Rejected. Worse than Option 1 under load.

---

### Option 3 — Saga Pattern with compensating transactions

Model the flow as a distributed saga. If Kafka publish fails, execute a compensating transaction to undo the order status change.

**Problems:**
- Significant complexity for a single-service scope
- Compensating transactions are hard to implement correctly
- The compensation for a failed Kafka publish would be reverting the order status — confusing for the customer
- No standard Spring Boot tooling supports this without adding Axon Framework or similar

**Verdict:** Rejected. Over-engineered for this scope.

---

### Option 4 — Debezium CDC (Change Data Capture)

Use Debezium to watch the PostgreSQL Write-Ahead Log (WAL) and automatically publish events to Kafka when rows change.

**How it works:**
- Debezium reads the PostgreSQL transaction log continuously
- Every committed INSERT or UPDATE triggers an event to Kafka automatically
- No application-level polling needed
- Guaranteed delivery because it reads from the WAL directly

**Problems:**
- Requires running a separate Debezium connector service
- Adds Kafka Connect infrastructure dependency
- Event payload shape is tied to DB column names — less control over payload structure
- Significant operational overhead for this assignment scope

**Verdict:** Rejected for this scope. This is the production-ideal evolution of the chosen approach and is documented in the README under "What I Would Do Differently."

---

### Option 5 — Transactional Outbox Pattern (Chosen)

Write the Kafka event payload to an `outbox_events` table in the same DB transaction as the order mutation. A separate scheduled component reads unpublished rows and publishes them to Kafka.

**Flow:**

```
Step 1 — Inside one @Transactional block:
  INSERT INTO orders (status = CONFIRMED, version = 1)
  INSERT INTO outbox_events (payload = {...}, published = false)
  Both commit atomically. If either fails, both roll back.

Step 2 — OutboxPoller runs every 5 seconds:
  SELECT * FROM outbox_events
  WHERE published = false
  FOR UPDATE SKIP LOCKED
  LIMIT 50

Step 3 — For each unpublished row:
  KafkaTemplate.send(topic, orderId, payload)
  Blocks until broker acknowledges (10 second timeout)

Step 4 — Only on successful Kafka send:
  UPDATE outbox_events SET published = true, published_at = now()

If Kafka send fails:
  Row stays published = false
  OutboxPoller retries on next cycle — no event is ever lost
```

**Why this works:**
- DB transaction guarantees atomicity of order + outbox writes
- If service crashes after DB commit, outbox row survives on disk
- On restart, OutboxPoller picks up unpublished rows automatically
- Kafka unavailability does not fail order placement
- Self-healing — no manual intervention needed

**Verdict:** Accepted.

---

## Decision

Implement the Transactional Outbox Pattern.

Every order state change writes an `outbox_events` row in the same `@Transactional` block as the order mutation. The OutboxPoller reads unpublished events every 5 seconds and publishes them to Kafka. Events are only marked published after a confirmed broker acknowledgement.

---

## Implementation

**Classes involved:**

| Class | Responsibility |
|---|---|
| `OrderTransactionalService` | Writes order + outbox event + idempotency key atomically |
| `OrderService` | Orchestrates status update + outbox event |
| `OutboxPoller` | Reads unpublished events, publishes to Kafka, marks done |
| `KafkaOrderEventPublisher` | Wraps KafkaTemplate with synchronous `.get()` send |

**Tables involved:**

| Table | Purpose |
|---|---|
| `orders` | Source of truth for order state |
| `outbox_events` | Queue of events waiting to be published to Kafka |

**Three dedicated save methods in `OrderTransactionalService`:**

- `persistOrder()` — saves order and items, own try-catch block
- `persistOutboxEvent()` — saves outbox event, own try-catch block
- `persistIdempotencyKey()` — saves idempotency key, handles race condition

Each method has targeted exception handling with dedicated log messages. When a failure occurs it is immediately clear which step failed and why. A `TransactionSynchronizationManager` callback is registered at transaction start — it logs `Transaction ROLLED BACK` explicitly if anything causes the transaction to fail, giving full visibility into rollbacks.

**Performance — partial index:**

```sql
CREATE INDEX idx_outbox_events_unpublished
    ON outbox_events (created_at ASC)
    WHERE published = false;
```

Only unpublished rows are indexed. Once published, rows fall out of the index. The poller query stays fast regardless of how many total rows the table contains.

**Multi-instance safety:**

The OutboxPoller query uses `FOR UPDATE SKIP LOCKED`. If multiple Order Service instances run simultaneously, each instance locks and processes a different batch of rows. No duplicate publishing occurs. This is critical for horizontal scaling — adding more instances increases throughput without any risk of double-publishing.

**Consumer-side deduplication:**

Kafka delivers at-least-once. The same event can arrive twice if the producer retries after a failed acknowledgement. The `delivery_notifications` table has a `UNIQUE` constraint on `event_id`. If the same event arrives twice, the second INSERT throws a `DataIntegrityViolationException` which the consumer catches and ignores silently.

**Status update idempotency:**

The assignment states the status update endpoint may be called up to 3 times with the same transition. `Order.transitionTo()` checks if current status equals requested status. If yes, it returns silently without changing state and without writing an outbox event. The same Kafka event is not published multiple times for the same transition.

---

## Trade-offs Accepted

| Trade-off | Why Accepted |
|---|---|
| Events published with up to 5 second delay | Eventual consistency is acceptable. Payment initiation and notifications do not require sub-second delivery. Order status is immediately correct in DB — only the downstream event has a short delay. |
| `outbox_events` table grows over time | Small cost compared to the consistency guarantee it provides. A cleanup job handles this in production. |
| At-least-once delivery — consumers must be idempotent | Handled via `event_id` UNIQUE constraint on `delivery_notifications`. |
| Polling adds small DB overhead every 5 seconds | Negligible. Query hits a partial index and returns quickly when no events are pending. |
| Debezium CDC would be lower latency | Debezium is the production evolution. Documented in README under "What I Would Do Differently." |

---

## What Would Be Done Differently in Production

**Replace OutboxPoller with Debezium CDC** — reads the PostgreSQL WAL directly and publishes to Kafka with near-zero latency and no DB polling overhead.

**Add outbox cleanup job** — a scheduled job to delete `outbox_events` rows older than the Kafka retention period to prevent unbounded table growth.

**Add monitoring on outbox lag** — alert if unpublished event count exceeds a threshold. A growing backlog means either Kafka is down or the poller has stopped. Expose this as a Micrometer metric on the `/actuator/prometheus` endpoint.
